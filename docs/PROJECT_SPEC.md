# 神州99 项目规范

## ⚠️ 铁律（不可违反）

### 1. 自测要求
**每次修改系统代码后，必须完成自测，通过后方可提交。**

- 修改任何 `src/` 下的代码 → 运行相关单元测试
- 修改策略逻辑 → 运行策略回测验证
- 修改风控逻辑 → 运行风控场景测试
- 修改交易接口 → 模拟盘验证
- 所有测试通过后才允许 merge / 部署
- CI/CD 流水线中必须包含自动化测试

### 2. 交易纪律
- **每笔交易必须设置止损**，无止损不允许开仓
- **盈亏比最低 1.5:1**，不符合的信号直接丢弃
- **单笔最大亏损不超过账户净值 2%**
- **日亏损超过 5% 暂停交易至次日**
- **最大回撤 10% 全部平仓并人工介入**

### 3. 风控优先
- 风控引擎拥有最高权限，可以否决任何交易信号
- 风控规则变更需双人审核
- 风控模块不可被策略模块绕过

## 代码规范

### Python 代码风格
- 遵循 PEP 8
- 类型注解（Type Hints）强制
- Docstring 使用 Google 风格
- 异步代码使用 async/await

### 命名规范
- 文件名：snake_case
- 类名：PascalCase
- 函数/方法：snake_case
- 常量：UPPER_SNAKE_CASE
- 私有方法/属性：前缀 `_`

### Git 规范
- 分支：`main` / `develop` / `feature/*` / `hotfix/*`
- Commit 消息格式：`type(scope): description`
  - type: feat / fix / refactor / test / docs / chore
  - 例：`feat(strategy): add trend following strategy`
- 每个 PR 必须有测试覆盖

### 配置管理
- 密钥（API Key/Secret）只存在 `.env` 文件中，绝不提交到 Git
- `.env.example` 提供模板
- 策略参数通过 `config/strategies.yaml` 配置，支持热更新

## 测试规范

### 测试层级
1. **单元测试：** 每个模块独立测试
2. **集成测试：** 模块间交互测试
3. **回测验证：** 策略在历史数据上的表现
4. **模拟盘测试：** OKX 模拟盘实时验证
5. **小资金实盘：** 最小资金实盘验证

### 覆盖率要求
- 核心模块（risk/、core/）：覆盖率 ≥ 90%
- 交易接口（exchange/）：覆盖率 ≥ 85%
- 策略模块（strategies/）：覆盖率 ≥ 80%

### 必须测试的场景
- [ ] 网络断连恢复
- [ ] API 限速处理
- [ ] 订单部分成交
- [ ] 极端行情（闪崩/暴涨）
- [ ] 同时多策略信号冲突
- [ ] 资金不足时的优雅降级
- [ ] 止损滑点超出预期

## 日志规范

### 日志级别
- **DEBUG：** 详细调试信息（仅开发环境）
- **INFO：** 关键操作（开仓、平仓、信号生成）
- **WARNING：** 风控触发、异常但可恢复
- **ERROR：** 下单失败、API 错误
- **CRITICAL：** 熔断触发、系统异常

### 交易日志必须记录
- 信号产生时间、策略名、方向、价格、理由
- 下单时间、订单ID、实际成交价、滑点
- 持仓变化、止损/止盈调整
- 平仓时间、盈亏金额、盈亏比
- 风控拒绝记录（拒绝原因）

## 部署规范

### 环境
- **开发环境：** 本地，使用 OKX 模拟盘
- **测试环境：** 服务器，OKX 模拟盘
- **生产环境：** 服务器，OKX 实盘

### 服务器要求
- 低延迟网络（距离 OKX 服务器近）
- UPS 或云服务器确保不断电
- 自动重启机制（systemd / supervisor）
- 监控告警（进程存活、API 连通性）

### 安全
- API Key 仅授予交易权限，不授予提现权限
- IP 白名单绑定
- 定期轮换 API Key
